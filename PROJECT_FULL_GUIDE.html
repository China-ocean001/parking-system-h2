💾 第二部分：数据库架构逐行深度解析 (schema.sql)

以下是对 src/main/resources/schema.sql 文件的逐行解释，帮助您彻底理解每一行 SQL 的作用。

Line 1-16: 环境清理 (Cleanup)
1-3 -- 注释说明：这是 H2 数据库脚本，用于每次启动重置环境
7-8 DROP VIEW IF EXISTS v_parking_details; -- 删除旧视图
9-15 DROP TABLE IF EXISTS ...; -- 删除旧表

详解：

  • 目的：因为使用 H2 内存数据库，为了防止开发过程中修改了表结构但旧数据残留导致报错，每次启动都先“清空现场”。
  • 顺序重要性：
    1. 先删视图 (View)：因为视图依赖于表。
    2. 再删子表 (Child Table)：如 parking_record（有外键）。
    3. 最后删父表 (Parent Table)：如 parking_space（被引用）。
Line 22-28: 用户表 (sys_user)
22 CREATE TABLE sys_user (
23     id BIGINT AUTO_INCREMENT PRIMARY KEY, -- 主键自增
24     username VARCHAR(50) NOT NULL,      -- 必填用户名
26     role VARCHAR(20) DEFAULT 'USER',    -- 默认为普通用户
27     create_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP -- 自动记录创建时间
28 );

详解： 系统所有用户（包括管理员）都存在这张表。role 字段决定了登录后是进入“后台管理”还是“用户预约”界面。

Line 31-36: 车位表 (parking_space)
31 CREATE TABLE parking_space (
33     space_number VARCHAR(20) NOT NULL, -- 如 "A001"
34     status INT DEFAULT 0,              -- 0:空闲 1:占用 2:预约
35     remark VARCHAR(200)                -- 如 "VIP车位"
36 );

详解： status 是系统的核心状态机。前端显示的绿色/红色/黄色方块完全取决于这个字段的值。

Line 39-49: 停车记录表 (parking_record)
39 CREATE TABLE parking_record (
43     entry_time TIMESTAMP NOT NULL,     -- 入场时间(必填)
44     exit_time TIMESTAMP,               -- 出场时间(先为空)
45     fee DECIMAL(10, 2),                -- 费用(保留2位小数)
48     FOREIGN KEY (space_id) REFERENCES parking_space(id) -- 外键关联
49 );

详解： 1. 流水表设计：每次进出产生一条新记录，而不是覆盖旧记录。 2. 费用固化：fee 字段在出场时计算并写入。这样做是为了审计——即使明年每小时单价涨了，也不会影响今年的历史账单金额。

Line 52-56: 免费车牌表 (free_plate)
52 CREATE TABLE free_plate (
54     plate_number VARCHAR(20) NOT NULL UNIQUE, -- 车牌必须唯一
56 );

详解： 简单的白名单表。业务逻辑：出场计费前先查此表，若存在则费用=0。

Line 59-68: 预约表 (reservation)
59 CREATE TABLE reservation (
65     status INT DEFAULT 0, -- 0:有效 1:已履约 2:已取消
66     FOREIGN KEY (user_id) REFERENCES sys_user(id),
67     FOREIGN KEY (space_id) REFERENCES parking_space(id)
68 );

详解： 关联了“谁”(`user_id`)预订了“哪里”(`space_id`)。入场时会检查：如果车位是预约状态，只有预约单上的车牌才能进。

Line 71-85: 风控模块
71 CREATE TABLE blacklist (...);       -- 黑名单：入场直接拦截
79 CREATE TABLE violation_record (...);-- 违规记录：仅做记录，累积多了进黑名单

详解： 这是系统的安全防线。黑名单表非常简单，只存车牌和原因，追求极速查询。

Line 91-99: 索引优化 (Performance) 重要
91 CREATE INDEX idx_record_plate ON parking_record(plate_number);
92 CREATE INDEX idx_blacklist_plate ON blacklist(plate_number);
96 CREATE INDEX idx_space_status ON parking_space(status);

详解：

  • idx_record_plate：车辆入场时，系统需要查询“这辆车是否有未支付的历史订单”。如果没有索引，系统需要扫描全表，速度极慢。
  • idx_space_status：首页显示“剩余车位”时，系统需要`SELECT * FROM space WHERE status=0`。索引能瞬间定位所有空闲车位。

Line 107-121: 实时详情视图 (v_parking_details) NEW
107 CREATE VIEW v_parking_details AS
108 SELECT r.id, s.space_number, r.plate_number...
118 FROM parking_record r
119 JOIN parking_space s ON r.space_id = s.id
120 LEFT JOIN free_plate fp ON r.plate_number = fp.plate_number
121 WHERE r.status = 0;

详解： 这是一个3表连接查询的封装。
管理员想看：“现在谁停在A001？停了多久？是不是免费车？”
不需要每次都写复杂的 JOIN SQL，直接查这个视图即可。WHERE r.status = 0 保证了只显示当前在场的车辆。

Line 124-131: 每日收入视图 (v_daily_income) NEW
124 CREATE VIEW v_daily_income AS
126 SELECT CAST(exit_time AS DATE), SUM(fee)
131 GROUP BY CAST(exit_time AS DATE);

详解： 利用 GROUP BY 预先计算每天的总收入。这是生成财务报表的利器。
